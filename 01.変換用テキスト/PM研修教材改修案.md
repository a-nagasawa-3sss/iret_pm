# PM研修教材改修案

対象ファイル：`高橋さん用フォルダ/PM実践研修模擬プロジェクトケース.md`
（ケース・スタディー「プロジェクトマネジャーの役割」／峰岸PMのケース）

参照ファイル（研修全体の構成・用語統一を確認するため）：
- `プロジェクトマネジメント実践研修_1.md` ～ `_6終結プロセス群.md`
- `プロジェクトマネジャーに 必要なヒューマンスキル.md`
- `交渉術ロールプレイ.md`
- `PM実践研修ケーススタディ（引っ越しプロジェクト）.md`

前提：本研修は現役PMではなく、**PMを目指す人向けの意識づけ・疑似体験**を目的とする。そのため誤字脱字や体裁だけでなく、「初心者が読んで気づきを得られるか」という観点を重視して確認した。

## 1. 誤字脱字・日本語文面の違和感 【対応済み】

- [x] 「本業との兼用ともあって疲れが出てきている」→「兼務」に修正（「兼用」は物・道具に使う語のため）。
- [x] 「峰岸課長をプロジェクトマネジャーとしてスタートしま／した。」→ PDF/Wordのページ折り返しがそのまま改行として残っていた箇所を「スタートしました。」に結合。
- [x] 「峰岸PMに対してかな／り批判的なものでした。」→ 同様に「かなり批判的な」に結合。
- [x] 「直接話しをせず」「話しをしてますが」「木村さんはIT言葉で話しをして」→ いずれも「話をする」表記に統一（3箇所）。
- [x] 「会議に必要な資料をお願いしても、ちゃんとルールに従ってよ。と冷たく言われてしまったり」→ カギ括弧で発言部分を明示（新田さんの発言全体を「」、その中の引用を『』の入れ子表記に修正）。
- [x] 「この２度ほど会議の席で」→「ここ２度ほど会議の席で」に修正。
- [x] 「アドバイザーとして不定期で、プロジェクトに参加している」→「不定期に」に修正。
- [x] ダブルクォート（`"…"`）で表記されていた地の文中の発言（前田さん・新田さん・佐々木課長のメール・峰岸PMの返信）をすべて日本語カギ括弧「」に統一。あわせて半角中点「･･」を全角「・・」に統一。
- [x] 【背 景】【課 題】の見出し内の余分な半角スペースを除去し、【背景】【課題】に修正。

　→ 全文チェックの結果、単語の途中で改行が分断されていたのは上記の2箇所（「スタートしま／した」「かな／り」）のみで、他の段落は文単位で改行されており分断はなかった。修正は `高橋さん用フォルダ/PM実践研修模擬プロジェクトケース.md` に直接反映済み。

## 2. 見出し・マークアップの適切性

- [x] **改行位置の全面見直し**：元がPDF等からの変換のため、文の途中（単語の切れ目でない場所）で改行されている箇所が散見される。全文を再チェックし、該当した2箇所（1.参照）を段落として結合済み。それ以外の行末は文単位（。／？／」）で終わっており分断は無かった。
- [x] 見出しレベルが `##`（H2）の直後に `####`（H4）へ飛んでおり、`###`（H3）が使われていなかった問題を修正。【登場人物】【背景】【課題】【問い】をすべて `####` → `###` に変更し、`##`（ケース全体タイトル）→`###`（各セクション）の正しい階層構成にした。
  　→ なお、この見出し飛び番の構成は他ファイル（`プロジェクトマネジメント実践研修_1.md`、`ヒューマンスキル.md`）にも共通して見られるため、フォルダ全体としては別途統一ルールの検討が望ましい（今回は対象ファイルのみ修正）。
- [x] 確認：【登場人物】は人物一覧のため末尾に半角スペース2つ（Markdownの改行記法）で改行、【背景】【課題】は地の文のため段落単位の改行、という違いは項目の性質（リスト vs 散文）によるもので問題無しと判断。
- [x] 工藤さんが伝えた4名（佐藤・佐々木・新田・前田）のコメントの「氏名：「発言」」形式は元々統一されていた。1.の対応でダブルクォート表記だった他の発言（前田さん・新田さん・佐々木課長のメール・峰岸PMの返信）もすべてカギ括弧に統一したため、文書全体で発言引用の形式が揃った。
- [x] 補足：他の教材ファイル（`交渉術ロールプレイ.md`、`プロジェクトマネジメント実践研修_2.md`）にある `~~誤~~正` の取り消し線表記は、既に対応済みの修正履歴であることを確認。本ファイルの修正では見せ消しは使わず、誤字は直接正しい表記に置き換えて反映した。

## 3. PMとしての素養が育つ構成になっているか

現状のケースは「ドキュメント作成に注力するあまりチーム内コミュニケーションが崩れていくPM」という、実務でも頻出する典型的な失敗パターンを扱っており題材として優れている。一方で、**受講者がPM未経験者であること**を踏まえると、以下の補強を提案する。

- [ ] **学習目標（ねらい）の明記が無い**：ケース本文の前に「このケースで学ぶこと」（例：資源マネジメント＝チームの育成、ステークホルダー・エンゲージメント、コミュニケーションマネジメントの重要性）を1〜2行で明示し、`プロジェクトマネジメント実践研修_1〜_6` で学んだ知識エリアと結び付ける導入文を追加する。
- [ ] **設問が単一で抽象的**：現状は「【問い】プロジェクトマネジャーの役割の観点から、このプロジェクトをどのように立て直しますか？」の1問のみ。PM未経験者には難易度が高いため、段階的な設問に分割することを推奨。
    - 例1）このプロジェクトで実際に起きている問題を整理してください（事実の整理）
    - 例2）その問題はなぜ起きたと考えられますか（原因分析：峰岸PMのマネジメントスタイル、コミュニケーション不足など）
    - 例3）PMBOK®の知識エリアで整理すると、どこに課題がありますか
    - 例4）峰岸PMは今後どう行動すべきですか（具体的なアクション）
- [ ] **ファシリテーター向け補助資料が無い**：想定される論点、参照すべきPMBOK®の知識エリア（資源／コミュニケーション／ステークホルダー）、模範的な振り返りポイントを別セクション（またはファシリテーター版として別ファイル）に用意すると、講師・受講者双方の理解を助ける。
- [ ] **登場人物が7名と多く関係が把握しづらい**：簡単な相関図（誰が誰の部下／誰と誰の関係が悪化しているか）を`[図解]`として追加すると、初心者でも状況を素早く把握できる。
- [ ] **ファイル名と内容のギャップ**：本ファイル名は「模擬プロジェクトケース」だが、内容は文書作成を伴わない議論主体のケーススタディである。一方「`PM実践研修ケーススタディ（引っ越しプロジェクト）.md`」は憲章・WBS・スケジュール等を実際に作成する演習であり、むしろ「模擬プロジェクト」という名称にふさわしい内容になっている。両ファイルの名称を入れ替える、または本ファイル名を「PM実践研修ケーススタディ（プロジェクトマネジャーの役割）.md」に変更するなど、名称と内容を一致させることを検討する。
- [ ] `プロジェクトマネジャーに 必要なヒューマンスキル.md` で扱われている「コミュニケーションの4原則」「ハイ/ローコンテクスト文化」等の概念と、本ケース内の登場人物間のすれ違い（新田と木村の関係、前田の不満など）を明示的に結び付ける参照コメントを追加すると、理論と疑似体験がつながりやすくなる。
